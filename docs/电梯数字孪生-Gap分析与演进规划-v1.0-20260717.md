## 1. 分析方法

Gap = 目标能力（规划 PR）−现状能力（规格 FR + 代码事实）。每条 Gap 五元组：现状（带证据）/ 目标（引 PR）/ 差距陈述 / 级别 / 关闭验收（破坏性，见 §5 各阶段表）。**关闭纪律：Gap 只在破坏性验收通过后关闭，代码合入不算。**

| 级别 | 含义 | 判据 |
|---|---|---|
| L1 品类硬伤 | 叫不叫得上"仿真"这个品类名 | 品类核心承诺不成立 |
| L2 任务阻断 | 核心任务走不通 | 任务成功条件缺前提 |
| L3 健壮性 | 静默失败 / 失效不可见 | 失效无法察觉或恢复 |
| L4 体验 | 可用但别扭 | 有绕行路径 |
| L5 工程债 | 用户不可见，拖慢演进 | 返工率 / 残留 / 不可测 |

## 2. 现状能力基线

### 2.1 已达标、应保持的能力

| 能力 | 说明（证据见附录 B） |
|---|---|
| 内存有界体系 | 乘客/日志/样本/批次全部封顶（B-14 等），长时运行不涨——优于多数教学向仿真 |
| GPU 资源纪律 | 场景重建全量 dispose（B-20），反复改配置不掉上下文 |
| 注入防护 | ext 模式 NaN/缺失字段回退（B-18），外部脏数据不会击穿渲染 |
| 满员语义 | 满员跳站 + passed_by 事件 + UI 提示（FR-2.4/2.5）——多数开源电梯 sim 无此实现 |
| 结构化日志 | 17 类 JSONL 事件（FR-8.2），带双时间戳，具备回放的原料条件 |
| 双语 i18n | 全 UI 中英切换即时生效（C-4） |
| 单文件交付 | 75KB 一个文件 scp 即上线（C-1），部署摩擦为零 |
| 布局持久化 | 拖拽/缩放/最小化存档+视口夹紧（FR-10.3，4b71b70 修复后） |

### 2.2 能力成熟度快照

| 域 | 评级 | 一句话判语 |
|---|---|---|
| 物理仿真 | ★★★★ | 运动学/门模/容量/负载驻留扎实——动得真 |
| 调度算法 | ★★★ | 三算法+满员+预泊；但 DCS 名实不符、不可插拔 |
| 指标体系 | ★★ | 11 项指标看得见，但口径自造、时基随倍速漂移——信不过 |
| 3D 表现 | ★★★★ | 建筑/轿厢/乘客/灯效完整；缺楼层与梯号标注 |
| 孪生接入 | ★★ | 写通道健壮；读取/订阅/回放全缺 |
| 工程质量 | ★★ | 内存与资源纪律好；零测试、零版本标识 |

总判语：**动得真、看得清，但数字信不过、结果带不走。**

## 6. 依赖与风险

| 风险 / 联动 | 影响 | 缓解 |

| CDN 兜底若走"内联 three"文件将 75KB→约 700KB | 加载性能与 C-1 观感 | 采用双源 fallback（本地 vendor 文件）而非内联 |
| 移动端适配与拖拽面板范式冲突——小屏上"可拖面板"根本不成立 | PR-P1-7 方向 | 方案定为断点切换单列折叠布局，非把拖拽搬上触屏 |
| 后台 rAF 节流不可消除 | 挂后台的长仿真 | 依赖 PR-P0-4 后冻结无害化，仅需提示（PR-P2-8） |
| 单人维护带宽 vs 五阶段范围 | 排期 | MoSCoW 纪律：P2 视资源、P3 默认不排期；每阶段可独立交付 |
