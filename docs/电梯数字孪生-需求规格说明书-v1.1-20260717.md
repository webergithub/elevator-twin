## 2. 产品概述

### 2.1 产品定位

面向演示、教学与集成验证的电梯群控数字孪生：在一个零安装的网页里同时呈现「物理仿真—AI 调度—三维孪生视图—运营指标」四层，并允许外部真实电梯系统经 JS API 接管孪生视图。核心场景：调度算法对比演示、运营 KPI 观察、外部系统孪生接入试验、电梯调度教学。

### 2.2 目标用户与典型场景

| 用户 | 典型场景 |
|---|---|
| 算法研究者 / 学生 | 切换 LOOK/NEAREST/DCS 与客流模式，观察 KPI 差异 |
| 展示 / 售前 | 大屏演示早高峰满载调度、满员跳站、待机预泊 |
| 集成开发者 | 用 window.DigitalTwin API 注入真实电梯状态验证孪生链路 |
| 站点访客 | OPC Studio 作品集浏览 |

### 2.3 设计约束与原则

| 编号 | 约束 / 原则 | 说明 |
|---|---|---|
| C-1 | 单文件零构建 | 全部功能在一个 index.html 内（846 行 / 约 75 KB），无打包步骤；发布即 scp 单文件 |
| C-2 | 运行时依赖仅 three CDN | importmap 引 three@0.160.0（cdn.jsdelivr.net），无其他外部依赖 |
| C-3 | 物理与渲染同环 | 仿真由 rAF 驱动，物理子步进上限 0.02s 保证高倍速数值稳定 |
| C-4 | 中英双语强制 | 全部 UI 文案走 i18n（data-zh/data-en + 字典 D），语言持久化 localStorage |
| C-5 | 内存有界 | 所有增长型缓冲一律设上限（详见附录 A），场景重建必须 dispose GPU 资源 |
| C-6 | OPC Studio 站点规范 | 顶部导航 44px、金色主题、返回主页链接 |

## 3. 系统架构

### 3.1 分层角色

| 层 | 角色 | 代码实体 |
|---|---|---|
| Layer 1 物理 | 单梯运动学、门状态机、群控派梯 | class EU / class ES |
| Layer 2 智能体 | 分时需求预测、待机位模拟退火优化、按钮统计 | class TA / IS / ElevatorAgent |
| Layer 3 孪生视图 | 建筑与轿厢三维渲染、外部状态跟随 | mkBld() / class E3D |
| Layer 4 数据 | 乘客生成（客流模板）、事件日志、KPI 统计 | genOD() / logEv() / updStats() |

### 3.2 数据流

sim 模式：genOD 批量生成乘客 → ES.call 派梯（算法三选一）→ EU 运动与开关门 → 事件（arr/dopen/exit…）→ 3D 视图 + 面板 + JSONL 日志。ext 模式：外部经 window.DigitalTwin.setElevState 写入状态 → E3D 与状态卡直接跟随外部值，内置物理停用（仅该轿厢视图层）。优化环：ElevatorAgent 每 30s（仿真时间）重算待机位 → ES.setIP → 空闲梯自动预泊（imv 事件）。

### 3.3 安全模型

纯前端静态页，无账号、无采集、无 Cookie；日志外发仅指向 localhost:8766 且在非 localhost 域名下被短路（需显式设 window.__ELV_LOG_SINK 才恢复）。全局 API 暴露于 window，供集成与调试，未做鉴权（演示品定位，见规划文档 §7 威胁建模）。

## 4. 功能需求

### 4.1 FR-1 配置与场景

| 编号 | 需求 |
|---|---|
| FR-1.1 | 楼层数滑杆 4–20（默认 10）、电梯数 1–5（默认 3）、批次间隔 2–15s（默认 5s），改动后点「应用配置」重建场景 |
| FR-1.2 | 调度算法下拉：LOOK 双向扫描（默认）/ 最近轿厢 / 目标控制 DCS，切换即时生效无需重建 |
| FR-1.3 | 客流模式下拉：自动（按时钟）/ 上行高峰 / 下行高峰 / 午餐双向 / 层间均匀，切换即时生效并写日志 |
| FR-1.4 | 动态参数折叠区：人数倍率加倍（×2 递增，上限 ×8）、倍率重置、每层生成权重滑杆 0–5（步进 0.5） |
| FR-1.5 | 全体紧急停梯 / 清除恢复（恢复时在梯乘客的目的层自动补回 stops，防止滞留） |
| FR-1.6 | 一键调度：清空 BOARDING 悬挂、强制空闲梯重新取目标 |
| FR-1.7 | 重置窗口布局按钮：恢复 4 列默认布局并立即持久化 |

### 4.2 FR-2 调度与群控

| 编号 | 需求 |
|---|---|
| FR-2.1 | LOOK：优先同向顺路梯（上行且梯在呼层下方 / 下行且在上方）取最近；无则取空闲梯最近；再无则全体最近 |
| FR-2.2 | NEAREST：全体非停运梯中取距呼层最近者 |
| FR-2.3 | DCS（简化实现）：取当前 stops 队列最短的梯（注意：非真实目的层分组，差距见 Gap 文档 G-SIM-6） |
| FR-2.4 | 满员跳站：三种算法的候选集合先排除满员梯（pax≥10）；若全部满员则退化为仅排除停运梯 |
| FR-2.5 | 满员开门时未能登梯的乘客回到 WAITING 并记 passed_by 事件，UI 提示「满员，N 人候下批」 |
| FR-2.6 | 空闲预泊：优化器输出各梯待机层，空闲且无任务的梯自动驶向待机层（imv 事件），距离阈值 0.5 层 |

### 4.3 FR-3 电梯物理

| 编号 | 需求 |
|---|---|
| FR-3.1 | 运动学：额定速度 2.5 层/s、加速度 1.8 层/s²，按制动距离判定减速点，到达判定 ε=0.03 层 |
| FR-3.2 | 门状态机：closed→opening(0.8s)→open(驻留)→closing(0.8s)→closed；关门中同层呼梯触发重开门（reopen 事件，剩余行程补偿） |
| FR-3.3 | 负载相关驻留：驻留 = min(6.0s, 1.6s + 下梯人数×0.75s)，每名登梯乘客再 +0.75s（上限同 6.0s）；重开门后基准重置为 1.6s |
| FR-3.4 | 单梯急停/恢复（状态卡按钮）与全体急停；急停梯不参与派梯 |
| FR-3.5 | 轿厢容量 10 人（CAP=10），board 超员被物理层拒绝 |
| FR-3.6 | 行程累计：每梯累计移动距离（层×4=米），启动次数计数（vel 0→非0） |

### 4.4 FR-4 乘客仿真

| 编号 | 需求 |
|---|---|
| FR-4.1 | 按批次生成：间隔 rate/倍率 秒，每批 1–3 基数×倍率（单批上限 8 人） |
| FR-4.2 | OD 生成按客流模式：auto 按本地时钟分段（7–9 时上行 / 17–19 时下行 / 12–13 时午餐 / 其余层间）；up=85% 从 1 层出发；down=85% 去往 1 层；lunch=大堂与中间层双向；inter=按每层权重均匀 |
| FR-4.3 | OD 重试上限 16 次 + 确定性兜底（fr%floors+1），单层权重独大时不会死循环 |
| FR-4.4 | 乘客状态机：WAITING→BOARDING（被派梯，走向梯口）→RIDING（隐藏）→EXITING（1.5s 淡出）→DONE（回收）；BOARDING 超 90s 自动回退 WAITING 重新派梯 |
| FR-4.5 | 排队可视化：同层等候者按 3 列×N 排站位；上限 120 人，超限静默移除最早的 WAITING（KI-2） |

### 4.5 FR-5 智能优化

| 编号 | 需求 |

## 6. 接口与协议规格

### 6.1 window.DigitalTwin（对外 JS API）

| 成员 | 方向 | 规格 |
|---|---|---|
| setElevState(id, st) | 写入 | st={floor:数值层号, dir:取 -1 / 0 / 1, doorPct:0–1, stops:数组}；字段可部分提供，缺失沿用上次；非有限数值回退默认（floor→1, dir→0, doorPct→0） |
| recordFloorCall(floor, dir) | 写入 | dir≥0 记上行；进入群控派梯与需求统计 |
| getRecommendation() | 读取 | 返回 {idlePos:各梯待机层数组, demand:归一化需求数组, pressCounts:按钮频次数组} |

### 6.2 JSONL 事件格式

每行一个 JSON 对象：公共字段 _ts（ISO8601）、_rel（相对启动秒）、t（域：取 elev / pax / opt / stat / sys / twin）、ev（事件名）；负载字段随事件而定（如 pax/arrive 含 pid/eid/f/wait_s）。

### 6.3 本地日志接收器

elevator-logger.py：HTTP POST localhost:8766/log，body={batch:[事件…]}，落盘 elevator.log；仅本地开发使用。
